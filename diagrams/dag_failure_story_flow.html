DAG Failure Handling Story

A simple management view of the current production support flow vs the AI-enhanced target state in GDC.

1) The Story

Today, the existing production workflow depends on scheduled jobs and orchestration across Apache DAGs, Control-M, and Spark submit execution. The process works functionally, but when a production DAG fails, the response is largely reactive. Geneos detects the issue and sends an alert email to the support team. From there, the support and development teams manually inspect logs, trace dependencies, and spend valuable time identifying the actual root cause before they can begin fixing the issue. This creates a delay between failure detection and meaningful resolution, especially when the logs are noisy, the failure is buried deep in execution details, or multiple teams need to collaborate.

As part of the GDC migration, the workflow is being modernized. Airflow is now orchestrating DAGs, with Trino and Iceberg supporting the data platform flow, and scheduled DAG execution continues without requiring separate Spark application submission in the same way as before. On top of this functional flow, an AI layer has been introduced using Gemma LLM running via Ollama. Now, when a DAG fails, the LLM is triggered automatically, reads the failure logs, analyzes the execution context, and produces a plain-English root cause analysis with probable fixes and additional technical clues. Instead of engineers starting from raw logs, they start with a guided diagnosis. This reduces mean time to identify issues, improves support efficiency, and helps teams move faster from alert to resolution.
Business impact: This changes the process from manual log hunting to AI-assisted troubleshooting, reducing debugging effort, speeding up response time, and making production support more consistent and scalable.
Current State

Existing workflow

  • Apache DAGs and Control-M coordinate production job execution.
  • Control-M performs Spark submit to run the application.
  • Jobs run on schedule and complete successfully in normal cases.
  • When a DAG fails in production, Geneos triggers an alert email.
  • Support team receives the alert and engages the development team.
  • Engineers manually inspect large logs and traces to find the root cause.
  • Fixing starts only after significant analysis effort.
Proposed AI-Enhanced State

Your solution in GDC

  • Airflow orchestrates DAGs in the new GDC-based data platform.
  • Trino and Iceberg are part of the runtime data ecosystem.
  • DAGs continue running on schedule in the target setup.
  • On DAG failure, Gemma LLM via Ollama is triggered automatically.
  • The model reads logs and execution details from the failed run.
  • It generates RCA in plain English with likely cause and possible fixes.
  • Support and dev teams start from guided insights instead of raw log exploration.
